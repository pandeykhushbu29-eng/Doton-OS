# Part I
# Foundation

# Chapter 1
# Goals

## 1.1 Purpose

The DESON Specification defines the observable behavior of a compiler
targeting the Doublers Instruction Set Architecture (ISA).

The purpose of this specification is to standardize compilation behavior,
compiler outputs, diagnostics, and compatibility while remaining
independent of implementation language, compiler architecture,
optimization strategy, or internal design.

This specification SHALL define what a compiler does,
not how it must be implemented.

---

## 1.2 Objectives

A conforming DESON implementation SHALL

• Read one or more source files.

• Validate lexical correctness.

• Validate syntax.

• Validate semantics.

• Report standardized diagnostics.

• Generate valid Doublers Assembly.

• Optionally generate object files.

• Optionally generate executable programs.

---

## 1.3 Scope

This specification defines

• Compiler behavior

• Source processing

• Lexical rules

• Parsing behavior

• Semantic validation

• Code generation

• Diagnostics

• Compatibility

This specification does NOT define

• IDEs

• Editors

• Debuggers

• Build systems

• Package managers

• Linkers

• Runtime libraries

• Optimizers

---

## 1.4 Design Philosophy

DESON follows five principles.

1.

Observable behavior is standardized.

2.

Internal implementation is unrestricted.

3.

Compatibility is more important than implementation.

4.

Compiler diagnostics should be understandable.

5.

Future language evolution shall preserve backwards compatibility whenever practical.

---

## 1.5 Terminology

Compiler

Software that transforms source code into another representation.

Compilation Unit

A single source file processed by the compiler.

Lexer

The component responsible for converting source text into tokens.

Parser

The component responsible for validating language grammar.

Semantic Analysis

Validation of program meaning beyond grammar.

Token

The smallest meaningful language element.

AST

Abstract Syntax Tree.

Code Generation

Transformation of validated programs into target output.

Diagnostic

A compiler-generated message describing warnings,
errors, or informational notes.

Executable

A runnable binary generated by the compiler.

Object File

A compiled but not yet executable program module.

Assembly

Human-readable instructions targeting the Doublers ISA.

---

## 1.6 Compiler Responsibilities

A conforming compiler SHALL

• Reject invalid source.

• Accept valid source.

• Produce deterministic output.

• Produce deterministic diagnostics.

• Preserve source ordering unless explicitly optimized.

---

## 1.7 Compiler Independence

The compiler MAY be implemented in

• Rust

• C

• C++

• Zig

• Java

• Python

• Go

• Any other programming language.

The implementation language SHALL NOT affect externally visible compiler behavior.

---

## 1.8 Compilation Pipeline

A conforming compiler SHALL conceptually perform

Source Files

↓

Lexical Analysis

↓

Parsing

↓

Semantic Analysis

↓

Code Generation

↓

Output Generation

Implementations MAY internally merge or split stages.

The externally visible result SHALL remain equivalent.

---

## 1.9 Input

The compiler SHALL accept

• One source file.

• Multiple source files.

• Standard input, if supported.

Source encoding SHOULD be UTF-8.

Implementations MAY support additional encodings.

---

## 1.10 Output

A compiler MAY produce

• Doublers Assembly

• Object Files

• Executables

• Debug Information

• Diagnostic Reports

Supported output formats SHALL be documented.

---

## 1.11 Errors

Compilation SHALL stop after unrecoverable errors.

Recoverable diagnostics MAY allow compilation to continue.

Fatal failures SHALL produce a non-zero exit code.

---

## 1.12 Warnings

Warnings SHALL NOT prevent compilation unless explicitly configured.

Warnings SHOULD describe

• potential mistakes

• deprecated language features

• unreachable code

• unused declarations

---

## 1.13 Compatibility

A compiler SHALL be considered DESON-compatible only if all mandatory requirements defined by this specification are implemented.

Optional features SHALL NOT modify mandatory compiler behavior.

---

## 1.14 Future Compatibility

Unknown future language constructs SHOULD generate clear diagnostics.

Implementations SHOULD ignore implementation-specific metadata whenever practical.

---

## 1.15 Summary

This chapter establishes the goals, scope, terminology, responsibilities,
and compatibility requirements of the DESON compiler.

Subsequent chapters define the observable behavior required for
conforming implementations.

# Chapter 2
# Compilation Model

## 2.1 General

Compilation is the process of transforming valid DESON source code into
one or more target outputs.

A conforming compiler SHALL perform every required validation before
producing executable output.

---

## 2.2 Compilation Unit

A compilation unit is a single source file.

Each compilation unit SHALL be processed independently before project
integration.

---

## 2.3 Project Compilation

A project MAY contain one or more compilation units.

The compiler SHALL compile every valid source file.

Files containing errors SHALL generate diagnostics.

---

## 2.4 Compilation Stages

The conceptual compilation pipeline is

Source

↓

Lexer

↓

Parser

↓

Semantic Analysis

↓

Intermediate Representation (optional)

↓

Code Generation

↓

Assembly

↓

Object File (optional)

↓

Executable (optional)

Implementations MAY internally reorganize these stages provided the
observable result remains equivalent.

---

## 2.5 Compilation Modes

A conforming compiler SHOULD support

• Assembly Generation

• Object Generation

• Executable Generation

Additional modes MAY be implemented.

---

## 2.6 Source Ordering

Compilation SHALL preserve source order whenever program semantics
depend on declaration order.

---

## 2.7 Multi-file Compilation

Multiple source files MAY be compiled as a single project.

The compiler SHALL resolve references between compilation units.

---

## 2.8 Compilation Failure

Compilation SHALL fail when

• lexical errors exist

• syntax errors exist

• semantic errors exist

• output generation fails

---

## 2.9 Compiler Exit Codes

0

Compilation successful.

1

Compilation failed.

Implementations MAY define additional exit codes.

---

## 2.10 Deterministic Output

Given identical

• source files

• compiler version

• compiler options

the compiler SHALL produce equivalent output.

---

## 2.11 Output Files

Generated filenames are implementation-defined.

Typical outputs include

program.asm

program.obj

program.exe

---

## 2.12 Compiler Configuration

Configuration MAY be supplied by

• command-line arguments

• project files

• configuration files

Implementations SHALL document supported configuration mechanisms.

---

## 2.13 Summary

This chapter defines the conceptual compilation model used throughout
the remainder of this specification.

---

# Chapter 3
# Source Language

## 3.1 General

The DESON source language defines the textual representation accepted by
the compiler.

The exact programming language syntax is defined by this specification.

---

## 3.2 Source Encoding

UTF-8 SHOULD be supported.

Additional encodings MAY be supported.

---

## 3.3 Source Files

Recommended extension

.des

Implementations MAY recognize additional extensions.

---

## 3.4 Whitespace

Whitespace SHALL separate language tokens where required.

Multiple spaces SHALL be treated as one separator.

Blank lines SHALL be ignored.

---

## 3.5 Comments

Single-line comments begin with

//

Everything following the comment marker SHALL be ignored.

Example

// Hello

print("Hello")

Multi-line comments MAY be supported.

---

Identifiers SHALL

• begin with a letter or underscore

• contain letters

• digits

• underscores

_main

counter1

Valid

Examples

my name

Invalid

---

## 3.7 Keywords

Reserved keywords SHALL NOT be used as identifiers.

Examples

else

while

return

function

const

var

true

false

null

---

## 3.8 Literals

The language SHALL recognize

Integer

Floating Point

Boolean

String

Character

Null

---

## 3.9 String Literals

Strings SHALL be enclosed using quotation marks.

Example

"Hello"

Escaped characters SHALL be supported.

---

## 3.10 Numeric Literals

Supported forms

3.14

-12

5e6

---

## 3.11 Boolean Literals

Supported values

false

---

## 3.12 Line Length

---

This chapter defines the basic textual structure accepted by DESON.

# Chapter 5
# Syntax

## 5.1 General

The parser SHALL validate that the token stream conforms to the DESON
language grammar.

Only lexically valid tokens SHALL be parsed.

---

## 5.2 Program Structure

A source file MAY contain

• Imports

• Constants

• Variables

• Structures

• Functions

• Comments

Declarations MAY appear in any order unless explicitly restricted.

---

## 5.3 Entry Point

A program SHALL contain exactly one entry function.

Example

function main()

{

    return 0;

}

Programs without an entry point SHALL fail compilation.

---

## 5.4 Statements

The language SHALL recognize

• Variable declarations

• Assignments

• Function calls

• Conditional statements

• Loops

• Return statements

• Empty statements

---

## 5.5 Variable Declaration

Example

var score = 10;

Variables SHALL have unique names within the same scope.

---

## 5.6 Constant Declaration

Example

const PI = 3.14159;

Constant values SHALL NOT be modified after initialization.

---

## 5.7 Assignment

Example

score = score + 1;

Assignments SHALL target writable objects.

---

## 5.8 Functions

Functions SHALL contain

• Name

• Parameters (optional)

• Return type (optional)

• Body

Example

function hello()

{

    print("Hello");

}

---

## 5.9 Function Calls

Example

hello();

print(score);

Function arguments SHALL match the declared parameter count unless
implementation-defined variadic functions are supported.

---

## 5.10 Conditional Statements

Example

if(score > 100)

{

    print("Winner");

}

else

{

    print("Try Again");

}

---

## 5.11 Loops

Supported loop forms SHALL include

while

for

Example

while(i < 10)

{

    i = i + 1;

}

---

## 5.12 Return Statements

Example

return 0;

A return statement SHALL terminate execution of the current function.

---

## 5.13 Blocks

Blocks SHALL be enclosed using

{

}

Nested blocks SHALL establish nested scopes.

---

## 5.14 Summary

The parser SHALL reject any source violating the grammar defined by this
specification.

---

# Chapter 6
# Semantic Analysis

## 6.1 General

Semantic analysis validates the meaning of syntactically correct programs.

Programs passing syntax validation MAY still fail semantic validation.

---

## 6.2 Scope

Each block SHALL introduce a new scope.

Identifiers declared inside a scope SHALL NOT be visible outside it.

---

## 6.3 Name Resolution

Every referenced identifier SHALL resolve to exactly one declaration.

Unknown identifiers SHALL generate diagnostics.

---

## 6.4 Duplicate Definitions

Duplicate identifiers within the same scope SHALL be rejected.

Example

var x = 1;

var x = 2;

Compilation SHALL fail.

---

## 6.5 Type Checking

Assignments SHALL use compatible types.

Example

Integer

↓

Integer

Valid

Integer

↓

String

Invalid

unless explicitly converted.

---

## 6.6 Constant Validation

Constants SHALL

• be initialized

• never be reassigned

Violation SHALL produce a semantic error.

---

## 6.7 Function Validation

The compiler SHALL verify

• function existence

• parameter count

• parameter types

• return compatibility

---

## 6.8 Return Validation

Functions declaring a return value SHALL return a compatible value along
every execution path.

---

## 6.9 Reachability

The compiler SHOULD detect unreachable code.

Example

return 0;

print("Hello");

The print statement SHOULD generate a warning.

---

## 6.10 Initialization

Variables SHOULD be initialized before use.

Reading an uninitialized variable SHOULD generate a warning.

---

## 6.11 Recursion

Recursive functions MAY be supported.

Implementations SHALL document recursion limitations, if any.

---

## 6.12 Compile-time Evaluation

Implementations MAY evaluate constant expressions during compilation.

Example

const SIZE = 5 + 10;

↓

15

---

## 6.13 Semantic Errors

Typical semantic errors include

UNKNOWN_IDENTIFIER

TYPE_MISMATCH

INVALID_ASSIGNMENT

INVALID_RETURN

DUPLICATE_IDENTIFIER

CONSTANT_MODIFICATION

INVALID_FUNCTION_CALL

---

## 6.14 Summary

Only programs that pass semantic validation SHALL proceed to code
generation.

# Chapter 7
# Code Generation

## 7.1 General

Code generation transforms semantically valid programs into executable
target representations.

Only semantically valid programs SHALL proceed to this stage.

---

## 7.2 Target ISA

The standard target architecture for DESON is

Doublers ISA

Future implementations MAY support additional targets.

---

## 7.3 Generated Output

A compiler SHALL generate one or more of

• Doublers Assembly

• Object File

• Executable

Supported formats SHALL be documented.

---

## 7.4 Instruction Selection

The compiler SHALL generate instructions that preserve the observable
behavior of the source program.

Instruction selection is implementation-defined.

---

## 7.5 Registers

Register allocation is implementation-defined.

Generated programs SHALL follow the Doublers ISA register conventions.

---

## 7.6 Memory Access

Generated code SHALL access memory only through valid instructions
defined by the target ISA.

---

## 7.7 Function Generation

Each source function SHALL generate one executable function.

Functions SHALL

• preserve call conventions

• preserve return behavior

• preserve parameter ordering

---

## 7.8 Variables

Variables MAY be stored

• in registers

• on the stack

• in static memory

Storage strategy is implementation-defined.

---

## 7.9 Constants

Compile-time constants SHOULD be embedded directly whenever practical.

---

## 7.10 Optimization

Optimizations MAY be performed.

Optimizations SHALL NOT alter observable program behavior.

---

## 7.11 Dead Code

Unused instructions MAY be removed.

Removal SHALL NOT affect externally visible behavior.

---

## 7.12 Code Ordering

Generated instruction order MAY differ from source order provided program
behavior remains equivalent.

---

## 7.13 Debug Information

Implementations MAY emit debugging information.

Debug formats are implementation-defined.

---

## 7.14 Code Generation Failure

Failures during generation SHALL terminate compilation.

No executable SHALL be produced after fatal generation errors.

---

## 7.15 Summary

The purpose of code generation is to produce valid executable output while
preserving the behavior of the source program.

---
